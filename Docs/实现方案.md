# 云原生微服务电商系统 - 实现方案

## Context

本项目为并行计算课程设计的电商微服务系统。现有3个Spring Boot后端微服务（user/cart/product）和一个单HTML前端，已通过Docker+Minikube完成基础部署。本方案将：
1. 将前端升级为React + Vite SPA
2. 新增订单微服务（加分项）
3. 新增JWT鉴权（加分项）
4. 部署到自有Linux服务器

## 一、整体架构

```
                    用户浏览器
                        |
                   Nginx(:80)
                  /  |  |  \  \
    静态资源  /users /cart /products /orders
                |      |      |        |
           user-svc cart-svc prod-svc order-svc
           (:8081)  (:8083) (:8082)  (:8084)
              |        |       |         |
           MySQL    Redis  MongoDB    MySQL
          (users)   (cart) (products) (orders)
                                (reviews)
```

**关键决策：**
- 订单服务复用现有MySQL实例（订单需要事务，且可减少资源占用）
- JWT无状态鉴权（微服务友好，无需额外Session存储）
- 前端构建为静态文件打入独立Docker镜像（替代ConfigMap挂载方式）
- 服务器部署使用Docker Compose（比K8s更轻量直观）

## 二、数据库设计

本系统采用混合持久化策略，根据各微服务的数据特点选择最合适的存储方案：

| 微服务 | 数据库 | 选型理由 |
|--------|--------|----------|
| user-service | MySQL | 用户数据需要事务（注册+余额初始化），结构化强，关系型最合适 |
| product-service | MongoDB | 商品属性差异大，需要动态Schema；评论与商品松耦合 |
| cart-service | Redis | 购物车高频读写，无需持久化到磁盘，Hash结构天然匹配 |
| order-service | MySQL | 订单需要ACID事务保证（创建订单+扣库存+扣余额） |

### 2.1 MySQL 设计（user-service / order-service）

**user-service — `users` 表：**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INT | PRIMARY KEY, AUTO_INCREMENT | 用户ID |
| username | VARCHAR(50) | UNIQUE, NOT NULL | 用户名 |
| password | VARCHAR(100) | NOT NULL | 密码（JWT阶段加@JsonIgnore不返回前端） |
| balance | DECIMAL(10,2) | DEFAULT 0.00 | 账户余额 |

**order-service — `orders` 表：**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PRIMARY KEY, AUTO_INCREMENT | 订单ID |
| user_id | INT | NOT NULL | 用户ID |
| total_price | DECIMAL(10,2) | NOT NULL | 订单总价 |
| status | VARCHAR(20) | NOT NULL, DEFAULT 'CREATED' | 状态: CREATED/PAID/SHIPPED/COMPLETED |
| created_at | DATETIME | NOT NULL | 创建时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |

**order-service — `order_items` 表：**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PRIMARY KEY, AUTO_INCREMENT | 订单项ID |
| order_id | BIGINT | NOT NULL, FK→orders.id | 所属订单 |
| product_id | INT | NOT NULL | 商品ID |
| product_name | VARCHAR(100) | NOT NULL | 商品名称（冗余快照） |
| price | DECIMAL(10,2) | NOT NULL | 下单时单价（冗余快照） |
| quantity | INT | NOT NULL | 购买数量 |

### 2.2 MongoDB 设计（product-service）

**选型依据：** 商品属性差异大（手机有颜色/存储/网络，键盘有轴体/背光），关系型数据库需要大量 NULL 列或 EAV 模式，MongoDB 的动态 Schema + 嵌入式文档天然适合这种场景。评论与商品松耦合，独立集合便于独立查询和扩展。

**`products` 集合 — 文档结构：**
```json
{
  "_id": 1,
  "name": "华为 Mate 60",
  "price": 6999.00,
  "stock": 50,
  "category": "Electronics",
  "attributes": {
    "color": "雅川青",
    "storage": "512GB",
    "network": "5G"
  }
}
```
- `_id`：Integer 类型，手动指定（非 ObjectId），便于跨服务引用
- `attributes`：嵌入式文档，不同品类商品可存放完全不同的属性键值对，这是 MongoDB 的核心优势

**`reviews` 集合 — 文档结构：**
```json
{
  "_id": "66a1b2c3d4e5f6a7b8c9d0e1",
  "productId": 1,
  "username": "张三",
  "content": "非常好用，推荐购买！",
  "rating": 5
}
```
- `_id`：String 类型，MongoDB 自动生成 ObjectId
- `productId`：关联 products 集合的 `_id`

**索引策略：**
- `reviews` 集合在 `productId` 字段上建索引，加速按商品查评价：`db.reviews.createIndex({productId: 1})`

**Spring Data MongoDB 映射：**
- `@Document(collection = "products")` 映射集合
- `@Id` 映射 `_id` 字段
- 动态字段 `Map<String, Object> attributes` 自动序列化为嵌套 BSON 文档

### 2.3 Redis 设计（cart-service）

**选型依据：** 购物车属于高频读写场景，数据无需持久化到磁盘（丢失可恢复），数据结构简单（用户→商品→数量的映射），Redis 基于内存的操作能满足低延迟要求。

**Key 设计：**

| 项目 | 说明 |
|------|------|
| Key 格式 | `cart:user:{userId}` — Hash 类型 |
| Hash Field | `{productId}` |
| Hash Value | `{quantity}` |
| 示例 | `cart:user:1` → `{ "1": 2, "3": 1 }` 表示用户1购物车有2件商品1、1件商品3 |

**序列化方案（对应 `RedisConfig.java`）：**
- Key 序列化：`StringRedisSerializer`
- Value / HashValue 序列化：`Jackson2JsonRedisSerializer<Object>`
- HashKey 序列化：`StringRedisSerializer`

**TTL 策略：** 可选设置 Key 过期时间为 7 天（`redisTemplate.expire(key, 7, TimeUnit.DAYS)`），避免僵尸购物车数据占用内存。

**操作与 Redis 命令对照：**

| 服务方法 | Redis 命令 | 说明 |
|----------|-----------|------|
| `addToCart(userId, productId, qty)` | `HSET cart:user:{userId} {productId} {qty}` | 添加/更新商品数量 |
| `getCart(userId)` | `HGETALL cart:user:{userId}` | 获取用户整个购物车 |
| `removeFromCart(userId, productId)` | `HDEL cart:user:{userId} {productId}` | 移除单个商品 |
| `clearCart(userId)` | `DEL cart:user:{userId}` | 清空整个购物车 |

## 三、统一响应格式

### 3.1 设计目标

前后端分离架构下，统一响应格式是微服务与前端契约的核心。本系统设计目标：
- 所有接口返回相同 JSON 结构，前端只需一套解析逻辑
- 业务状态码与 HTTP 状态码分离：HTTP 状态码表示传输层结果（200=到达服务器），业务 code 表示业务层结果
- 错误信息对用户友好、对开发者可调试

### 3.2 响应体结构

```java
public class ApiResponse<T> {
    private int code;       // 业务状态码
    private String message; // 提示信息（成功时为 "success"，失败时为可读错误描述）
    private T data;         // 业务数据（失败时为 null）
    private long timestamp; // 响应时间戳（毫秒级，便于前端调试和缓存判断）

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(200, "success", data);
    }
    public static <T> ApiResponse<T> success() {
        return new ApiResponse<>(200, "success", null);
    }
    public static <T> ApiResponse<T> error(int code, String message) {
        return new ApiResponse<>(code, message, null);
    }
}
```

### 3.3 业务状态码规范

| 状态码 | 含义 | 典型场景 |
|--------|------|----------|
| 200 | 成功 | 查询成功、创建成功、更新成功、删除成功 |
| 400 | 参数错误/业务校验失败 | 用户名已存在、密码错误、购物车为空、库存不足 |
| 401 | 未认证 | 未携带Token、Token过期、Token无效 |
| 403 | 无权限 | 用户试图操作他人数据（预留，当前未启用） |
| 404 | 资源不存在 | 商品不存在、订单不存在、用户不存在 |
| 500 | 服务器内部错误 | 未捕获异常、数据库连接失败等 |

**查询成功（有数据）：**


**业务校验失败：**


**未认证：**


### 3.5 全局异常处理器

各微服务通过 `@RestControllerAdvice` 统一捕获异常并转换为 `ApiResponse`，避免 Controller 中大量 try-catch：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 业务异常（如「用户名已存在」「库存不足」）
    @ExceptionHandler(BusinessException.class)
    public ApiResponse<?> handleBusiness(BusinessException e) {
        return ApiResponse.error(e.getCode(), e.getMessage());
    }

    // 参数校验异常
    @ExceptionHandler(IllegalArgumentException.class)
    public ApiResponse<?> handleIllegalArg(IllegalArgumentException e) {
        return ApiResponse.error(400, e.getMessage());
    }

    // 未捕获异常兜底
    @ExceptionHandler(Exception.class)
    public ApiResponse<?> handleException(Exception e) {
        log.error("未捕获异常", e);
        return ApiResponse.error(500, "服务器内部错误，请稍后重试");
    }
}
```

其中 `BusinessException` 为自定义业务异常类：
```java
public class BusinessException extends RuntimeException {
    private final int code;
    public BusinessException(int code, String message) {
        super(message);
        this.code = code;
    }
}
```

### 3.6 分页响应格式

对于列表查询接口（如 `GET /products`），`data` 字段采用以下分页结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "content": [ ... ],
    "page": 0,
    "size": 10,
    "totalElements": 25,
    "totalPages": 3
  },
  "timestamp": 1752105600000
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| content | Array | 当前页数据列表 |
| page | int | 当前页码（从 0 开始） |
| size | int | 每页条数 |
| totalElements | long | 总记录数 |
| totalPages | int | 总页数 |

### 3.7 前端处理约定

前端 Axios 响应拦截器统一处理逻辑：
```javascript
axios.interceptors.response.use(
  response => {
    const { code, message, data } = response.data;
    if (code === 200) return data;        // 成功：直接返回业务数据
    if (code === 401) {                   // 未认证：跳转登录页
      localStorage.removeItem('token');
      window.location.href = '/login';
      return Promise.reject(new Error(message));
    }
    return Promise.reject(new Error(message)); // 其他错误：抛出供 UI 展示
  },
  error => {
    return Promise.reject(new Error('网络异常，请检查连接'));
  }
);
```

### 3.8 改造要求

各微服务所有 Controller 方法返回类型统一为 `ResponseEntity<ApiResponse<T>>`，不再直接返回实体、字符串或 Map。配合全局异常处理器，Controller 代码只需关注业务逻辑，无需手动 try-catch。

## 十、注意事项

- **Minikube内存**：新增服务后共10个Pod，建议order-service JVM限制 `-Xmx256m`

- **React Router 404**：Nginx必须配置 `try_files $uri $uri/ /index.html`
- **跨服务调用**：K8s和Docker Compose中服务名均为 `cart-service` 等，无需改代码
