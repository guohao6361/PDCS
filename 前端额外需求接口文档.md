## 1. 支付订单

**请求方式：** `POST`

**请求路径：** `/orders/{id}/pay`

**路径参数：**
| 参数名 | 类型 | 说明 |
|--------|------|------|
| id | Long | 订单ID |

**请求头：**
| 参数名 | 类型 | 说明 |
|--------|------|------|
| Authorization | String | Bearer {JWT Token} |

**响应示例：**
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 1,
    "userId": 1,
    "totalPrice": 299.00,
    "status": "PAID",
    "createdAt": "2026-07-11T10:00:00"
  }
}
```

**业务逻辑：**
- 将订单状态从 `CREATED` 更新为 `PAID`
- 扣减用户余额
- 仅 `CREATED` 状态的订单可支付

## 2.1 订单支付时限（15 分钟超时自动取消）

**需求描述：**  
订单创建后，用户需在 15 分钟内完成支付，超时未支付则自动取消。

**业务规则：**
- 订单创建时记录 `createdAt` 时间戳
- 前端倒计时 15 分钟，超时后自动调用 `POST /orders/{id}/cancel` 取消订单
- 后端存储订单信息，订单数据从后端读取，不得仅存前端
- 后端也应校验时限，超过 15 分钟的未支付订单拒绝支付

**前端交互：**
- 订单详情页/支付页显示倒计时（分:秒）
- 倒计时归零时提示「订单已超时，已自动取消」并跳转订单列表
- 倒计时基于后端 `createdAt` 计算，不依赖前端本地时间

**当前问题：**  
- 后端订单信息未完整存储
- 无超时自动取消机制

**当前状态：** ❌ 后端未实现订单超时取消逻辑

## 3. 商品分批加载（分页查询）

**需求描述：**  
前端商品列表页应分批请求商品数据，而非一次性全量加载，以提升页面加载速度和用户体验。

**后端接口（已实现）：** `GET /products?page={page}&size={size}`

**请求参数：**
| 参数名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| page | int | 0 | 页码（从0开始） |
| size | int | 10 | 每页数量 |

**响应示例：**
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "content": [
      { "id": 1, "name": "商品A", "price": 99.00, "category": "电子" }
    ],
    "totalElements": 50,
    "totalPages": 5,
    "number": 0,
    "size": 10,
    "first": true,
    "last": false
  }
}
```

**前端实现要求：**
- 首次加载请求第 0 页（如 8 条）
- 滚动到底部或点击「加载更多」时请求下一页
- 根据 `last` 字段判断是否还有更多数据
- 搜索过滤仅对已加载数据进行客户端过滤

**当前状态：** ✅ 前端已实现分批加载（首次加载 8 条，点击「加载更多」请求下一页）

## 7. 余额扣减逻辑修复（避免重复扣款）

**当前问题：**  
前端在购物车结算创建订单时，会立即扣减用户余额：
```javascript
// Cart.jsx
updateUser({ ...user, balance: user.balance - order.totalPrice });
```
但订单支付时后端也会再次扣减余额，导致**同一笔订单扣款两次**。

**正确流程：**  
| 操作 | 余额变化 |
|------|----------|
| 创建订单（CREATED） | ❌ 不扣余额 |
| 支付订单（PAID） | ✅ 扣减余额 |
| 取消订单（CANCELLED） | ❌ 不扣余额 |

**修复要求：**  
1. **前端**：创建订单时**不得**在前端扣减余额，移除 `updateUser({ ...user, balance: ... })` 逻辑
2. **后端**：余额扣减仅在 `POST /orders/{id}/pay` 支付接口中执行
3. 支付成功后，前端调用 `GET /users/{id}` 刷新余额

**当前状态：** ❌ 前端存在重复扣款问题，需修复

## 9. 密码管理（登录密码 + 支付密码）

**需求描述：**  
用户需拥有两套密码：登录密码和支付密码，分别用于身份验证和支付确认。

**密码定义：**
| 密码类型 | 用途 | 长度要求 |
|----------|------|----------|
| 登录密码 | 用户登录系统 | 6-20 位 |
| 支付密码 | 支付订单时验证身份 | 6 位数字 |

**后端接口需求：**

### 9.1 修改登录密码

**接口：** `PUT /users/{id}/password`

**请求体：**
```json
{
  "oldPassword": "old123456",
  "newPassword": "new123456"
}
```

**请求头：**
| 参数名 | 类型 | 说明 |
|--------|------|------|
| Authorization | String | Bearer {JWT Token} |

### 9.2 修改支付密码

**接口：** `PUT /users/{id}/pay-password`

**请求体：**
```json
{
  "oldPayPassword": "123456",
  "newPayPassword": "654321"
}
```

**请求头：**
| 参数名 | 类型 | 说明 |
|--------|------|------|
| Authorization | String | Bearer {JWT Token} |

### 9.3 重置密码（忘记密码）

**接口：** `POST /users/{id}/reset-password`

**请求体：**
```json
{
  "type": "LOGIN",
  "newPassword": "new123456"
}
```

**说明：** `type` 可为 `LOGIN`（登录密码）或 `PAY`（支付密码）

### 9.4 支付时验证支付密码

**接口：** `POST /orders/{id}/pay`

**请求体（修改）：**
```json
{
  "payPassword": "654321"
}
```

**业务逻辑：**
- 支付时需校验支付密码正确性
- 支付密码错误则拒绝支付

### 9.5 注册时设置双密码

**接口：** `POST /users/register`

**请求体（修改）：**
```json
{
  "username": "user1",
  "password": "login123",
  "payPassword": "123456"
}
```

**当前问题：** 后端 User 实体仅有 `password` 字段，缺少 `payPassword` 字段

**当前状态：** ❌ 后端未实现双密码体系，无密码修改/重置接口

## 13. 订单状态扩展与管理

**需求描述：**  
当前订单仅有 3 个状态（CREATED/PAID/CANCELLED），需扩展为完整的物流状态机。

**新状态定义：**
| 状态 | 标识 | 说明 |
|------|------|------|
| 未支付 | `UNPAID` | 订单已创建，待支付 |
| 已支付 | `PAID` | 已支付，待发货 |
| 已发货 | `SHIPPED` | 商家已发货 |
| 运输中 | `IN_TRANSIT` | 包裹运输中 |
| 已送达 | `DELIVERED` | 包裹已送达 |
| 已完成 | `COMPLETED` | 用户确认收货 |
| 已取消 | `CANCELLED` | 订单已取消 |

**状态流转图：**
```
UNPAID → PAID → SHIPPED → IN_TRANSIT → DELIVERED → COMPLETED
  ↓
CANCELLED
```

**后端接口需求：**

### 13.1 更新订单状态

**接口：** `PUT /orders/{id}/status`

**请求体：**
```json
{
  "status": "SHIPPED"
}
```

**请求头：**
| 参数名 | 类型 | 说明 |
|--------|------|------|
| Authorization | String | Bearer {JWT Token} |

**权限控制：**

| 状态变更 | 操作角色 | 说明 |
|----------|----------|------|
| UNPAID → PAID | 用户 | 用户支付订单 |
| UNPAID → CANCELLED | 用户 | 用户在未支付时取消 |
| PAID → CANCELLED | 用户 | 用户在已支付时取消 |
| DELIVERED → COMPLETED | 用户 | 用户确认收货 |
| PAID → SHIPPED | 商家 | 商家发货 |
| 任意状态变更 | 管理员 | 管理员可修改任意订单状态 |

### 13.2 前端展示要求

- 订单列表显示完整状态中文名称
- 不同状态用不同颜色区分
- 用户可按状态筛选订单
- 商家可操作「发货」按钮
- 用户可对已送达订单操作「确认收货」

**当前状态：** ❌ 后端仅有 3 个状态，缺少物流状态流转接口

**需求描述：**

**地址字段：**
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|

| phone | String | 是 | 收件人手机号 |
| province | String | 是 | 省 |
| city | String | 是 | 市 |
| district | String | 是 | 区 |
| detailAddress | String | 是 | 详细地址 |
| isDefault | Boolean | 否 | 是否默认地址 |

**后端接口需求：**

### 14.1 获取用户地址列表

```json

  "code": 200,
  "data": [
    {
      "id": 1,
      "receiverName": "张三",
      "phone": "13800138000",
      "province": "广东省",
      "city": "深圳市",
      "district": "南山区",
      "detailAddress": "xx路xx号",
      "isDefault": true
    }
  ]
}
```

### 14.2 新增收货地址

**接口：** `POST /users/{id}/addresses`

**请求体：**
```json
{
  "receiverName": "张三",
  "phone": "13800138000",
  "province": "广东省",
  "city": "深圳市",
  "district": "南山区",
  "detailAddress": "xx路xx号",
  "isDefault": true
}
```

**业务规则：**
- 每个用户最多 10 个地址，超出返回错误
- 设置 `isDefault=true` 时，自动将其他地址设为非默认

### 14.3 修改收货地址

**接口：** `PUT /users/{id}/addresses/{addressId}`

**请求体：** 同新增，支持部分更新

### 14.4 删除收货地址

**接口：** `DELETE /users/{id}/addresses/{addressId}`

### 14.5 设置默认地址

**接口：** `PUT /users/{id}/addresses/{addressId}/default`

### 14.6 支付订单时选择收货地址

**业务规则：**
- 支付订单时必须选择一个收货地址，订单中记录所选地址 ID
- 若用户无任何收货地址，强制引导先创建地址才能支付
- 已有地址时，也可在支付流程中新建地址
- 地址上限 10 个，达到上限时必须先删除已有地址才能新建

**前端交互流程：**
```
用户点击支付
  → 调用 GET /users/{id}/addresses 获取地址列表
  → 无地址：弹出「请先添加收货地址」，引导新建
  → 有地址：展示地址列表供选择，每个地址可点击「删除」，同时可点击「新增地址」
  → 已选地址后完成支付
```

**当前问题：**  
- 后端无地址实体和表
- 无地址 CRUD 接口
- 订单无收货地址字段，支付时无法选择地址
